7. 모델의 세 가지 갈래
브랜드 말고 성격으로 나눠라.
TYPE A
빠른 모델
짧은 프롬프트 → 빠른 답.
UI 시안·문구·간단 변환에 강함.
예: 경량·빠른 응답 계열
TYPE B
추론 모델
복잡한 로직·구조 설계.
오류 디버깅·아키텍처 결정에 강함.
예: 추론·고성능 코딩 계열
TYPE C
대중 모델
균형형, 일상 사용에 무난.
처음 입문자에게 가장 친숙.
예: ChatGPT·Claude·Gemini 기본형
전략: 빠른 작업엔 A, 어려운 일엔 B, 일상은 C. 이 세 가지를 상황별로 갈아탄다.

8.5 한눈에 보기
유형강점약점언제 쓸까
채팅형가장 쉬움. 대화로 바로 묻고 답함실제 결과를 화면으로 못 본다아이디어·기획·문구
캔버스형코드와 결과를 한 화면에서 본다큰 프로젝트엔 부적합한 화면 도구·프로토타입
에디터형여러 파일 다루기 편함설치·설정 부담여러 화면 앱·반복 작업
에이전트형스스로 파일 만들고 실행예상 밖 행동 위험익숙해진 뒤, 명확한 요구
통제력을 줄수록 자동화가 늘고, 자동화가 늘수록 검토가 더 중요해진다.

9. 단계별 추천 조합
자기 단계에 맞는 도구가 정답이다.
STAGE 1
처음 시작
한 화면짜리 도구를 만든다.
대중 모델채팅형캔버스형
→ 결과를 즉시 보고, 사이클을 익힌다.
STAGE 2
조금 익숙
여러 화면 도구·복수 파일.
추론 모델에디터형
→ 디버깅·구조 설계가 늘어난다.
STAGE 3
자동화
반복 작업을 위임한다.
추론 모델에이전트형
→ 명확한 경계와 검토가 필수.
L03 · 조합

9.4 빠른 시작 vs 전문 시작
처음부터 거창하게 세팅하지 않는다. 목적에 맞게 나눈다.
OPTION A · 빠른 시작
채팅 + 캔버스
설치 없이 오늘 바로 첫 화면을 띄운다. 동아리 첫 실습에 가장 적합하다.
목표한 화면 도구, HTML 초안, 간단한 자동화
도구ChatGPT / Claude / Gemini + Canvas·Artifacts
장점진입장벽 낮음, 결과 확인 빠름
주의큰 프로젝트·다중 파일 관리에는 한계
OPTION B · 전문 시작
VS Code/Cursor + GitHub + 배포
서비스처럼 운영할 작은 앱을 만들 때 선택한다. 구조·버전·배포를 함께 관리한다.
코딩VS Code, Cursor, Copilot, 에이전트형 도구
버전GitHub 저장소
백엔드Supabase 등 관리형 DB·Auth
배포Vercel / Firebase Hosting 등
동아리 1차 목표는 빠른 시작. 반복해서 만들 수 있게 되면 전문 시작으로 확장한다.

10. 흔한 세 가지 함정
TRAP 01
"가장 좋다"는 도구를 따라간다
자기 단계와 안 맞으면 의미가 없다. 가장 좋은 도구가 아니라 지금 내가 익힐 수 있는 도구가 답이다.
TRAP 02
한 도구로 모든 일을 한다
기획·디버깅·자동화는 성격이 다르다. 한 도구에 묶여 있으면 작업의 성격에 따라 갈아타는 유연함을 잃는다.
TRAP 03
도구가 다 해줄 거라 기대한다
도구는 도구일 뿐이다. 의도 설계와 결과 검토는 여전히 사람의 일이다.
도구는 정답이 아니라 선택지다.

13. 첫 프롬프트 템플릿
2강의 5대 원칙을 그대로 끼워 넣었다.
나는 [반복적으로 하는 작업]을 줄이기 위해 한 화면짜리 웹 도구를 만들고 싶다. 사용자는 [나 자신 / 동료]이고, IT 지식은 일반 수준이다. 화면을 열면 가장 먼저 [입력 필드]가 보여야 한다. 사용자가 [입력 행동]을 하면, [결과]가 화면 아래에 표시된다. 완료되면 결과를 복사할 수 있는 버튼이 함께 보인다. 문구·배치·색상은 알아서 정해도 된다. 하지만 새로운 화면이나 메뉴는 추가하지 마라. 복잡한 데이터 저장도 하지 마라. 이번에는 완성본이 아니라 1차 초안을 만들어라.
