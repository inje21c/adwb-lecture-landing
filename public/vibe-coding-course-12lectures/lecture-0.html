02 · OPENAI · 한도 설정 예시
처음에는 작게 잡고 늘린다.
SOFT LIMIT
5 USD
이 금액에 도달하면 알림 메일이 온다. 차단은 안 됨.
HARD LIMIT
10 USD
이 금액에 도달하면 API 호출이 자동 차단된다.
AUTO RECHARGE
OFF
자동 충전은 일단 끈다. 모니터링 습관이 먼저.
왜 두 개를 다 잡나? Soft만 잡으면 알람만 오고 계속 과금된다. Hard만 잡으면 미리 알 수 없다.

PS> cd C:\Users\me\keys PS> icacls vc-key.pem /inheritance:r PS> icacls vc-key.pem /grant:r "$($env:USERNAME):(R)" PS> ssh -i vc-key.pem ubuntu@<EC2 퍼블릭 IP> The authenticity of host ... can't be established. Are you sure you want to continue connecting? yes ubuntu@ip-172-31-...:~$ _
왜 icacls가 필요한가
.pem 파일이 너무 공개적이면 SSH가 거부한다. Linux의 chmod 400에 해당하는 윈도우 명령이 icacls.
사용자 이름
Ubuntu AMI는 ubuntu. Amazon Linux는 ec2-user. 헷갈리면 콘솔에서 "Connect" 버튼이 알려줌.

11 · .ENV · 비밀값을 코드에서 분리한다
키는 .env에만, 코드에는 참조만.
✗ 위험 — 코드에 직접 박기
const apiKey = "sk-proj-AbCd1234..."; fetch("https://api.openai.com/...", { headers: { "Authorization": "Bearer " + apiKey } });
→ 이 파일을 GitHub에 올리는 순간 키가 공개된다.
✓ 권장 — .env에서 읽기
// .env (절대 커밋하지 않음) OPENAI_API_KEY=sk-proj-AbCd1234... // app.js const apiKey = process.env.OPENAI_API_KEY;
→ 코드는 키를 모른 채 동작한다.

12 · .GITIGNORE · 올라가지 않게 막는다
GitHub에 절대 올라가면 안 되는 파일 목록.

왜 .pem도 넣나
EC2 키 파일은 서버 접속 비밀번호와 같은 무게. 실수로 올리면 누구나 본인 서버에 접속 가능.
올린 후 발견했다면
단순 삭제 + 커밋으로는 부족. 해당 키를 즉시 폐기하고 새로 발급한 뒤 히스토리 정리.
