LECTURE 08
왜 RFP까지 가는가
  • demo는 가능성을 보여준다
  • 기능정의서는 범위를 설명한다
  • RFP는 실행 조건과 책임을 정리한다
  • 발표는 이해관계자의 의사결정을 돕는다
비개발자의 바이브코딩은 demo에서 끝나지 않고, 조직 의사결정 산출물로 이어져야 한다.

LECTURE 08
기능 요구사항 변환
기능정의서RFP 문장
아이디어 입력사용자는 아이디어, 사용자, 문제, KPI, 제약 조건을 입력할 수 있어야 한다
산출물 생성시스템은 입력값을 기반으로 보도자료 문장, KPI, 화면 프롬프트, 구현 프롬프트를 생성해야 한다
결과 복사사용자는 생성된 결과를 항목별로 복사할 수 있어야 한다
초기화사용자는 입력값과 결과를 초기화할 수 있어야 한다

LECTURE 08
RFP 생성 프롬프트
다음 기능정의서를 바탕으로 제안요청서(RFP) 초안을 작성해줘. 포함할 항목은 다음과 같다. 1. 프로젝트 배경 2. 추진 목적 3. 대상 사용자 4. 과업 범위 5. 기능 요구사항 6. 비기능 요구사항 7. 보안 및 데이터 처리 요구사항 8. 산출물 9. 일정 및 수행 방식 10. 제안서 평가 기준 11. 제외 범위 문체는 기업 내부 검토용 RFP 초안처럼 공식적이고 간결하게 작성해라. 불확실한 부분은 임의로 단정하지 말고 '[확인 필요]'로 표시해라.
