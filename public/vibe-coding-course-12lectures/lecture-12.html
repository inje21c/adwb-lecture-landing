LECTURE 12
개인정보와 고객정보는 기본적으로 제외
금융회사나 고객데이터를 다루는 조직에서는 특히 보수적으로 접근해야 합니다.
  • 교육 실습에는 식별 가능한 개인정보를 쓰지 않는다
  • 고객명, 연락처, 계좌번호, 계약번호, 상담번호는 제외한다
  • 거래 내역과 신용 정보는 더 강한 통제가 필요하다
  • 필요하면 익명화·마스킹·집계 후 사용한다
  • 판단이 애매하면 사용하지 않는다

LECTURE 12
비밀값과 접근 통제
작은 도구라도 접근과 비밀값은 반드시 관리해야 합니다.
  • API Key를 클라이언트 코드에 넣지 않는다
  • 공개 저장소에 비밀값을 올리지 않는다
  • 배포 링크를 누구나 접근할 수 있게 열지 않는다
  • 팀 도구는 접근 범위와 책임자를 정한다
  • 실제 업무 적용은 사내 인증·권한 체계를 검토한다

LECTURE 12
변경 통제의 최소 원칙
작은 도구도 사용자가 생기면 변경 통제가 필요합니다.
  • 동작하는 버전을 기준으로 보관한다
  • 큰 변경은 한 번에 하나씩 한다
  • 변경 이유를 커밋 메시지나 변경 이력에 적는다
  • 사용자에게 영향을 주는 변경은 공지한다
  • 문제가 생기면 이전 버전으로 되돌린다

LECTURE 12
12강 전체 회고
전체 과정의 핵심을 다섯 문장으로 정리합니다.
  • 바이브코딩은 코드 작성 기술이 아니라 업무 문제 번역 능력이다
  • Working Backwards는 만들기 전에 결과를 선명하게 만든다
  • 화면과 demo는 요구사항을 빠르게 정렬한다
  • 데이터와 AI는 도구를 실무 흐름에 붙인다
  • 보안과 운영 기준이 있어야 회사 도구가 된다
