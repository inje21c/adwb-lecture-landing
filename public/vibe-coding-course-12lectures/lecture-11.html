GitHub는 코드 창고가 아니라 변경 이력이다
비개발자에게 GitHub는 협업과 기록의 도구로 설명하는 것이 좋습니다.
  • 처음 만든 버전을 저장한다
  • 수정할 때마다 기록을 남긴다
  • 문제가 생기면 이전 버전과 비교한다
  • README로 사용법과 제한사항을 설명한다
  • 동료 피드백을 이슈로 정리할 수 있다

LECTURE 11
절대 배포하면 안 되는 것
공유 실습은 교육용 샘플 버전만 대상으로 합니다.
  • API Key가 코드 안에 들어간 파일
  • 실제 고객정보가 포함된 샘플 데이터
  • 내부 실적 원본이나 비공개 문서
  • 회사 계정 또는 인증 우회 정보
  • 외부 공개가 불가능한 화면 캡처

LECTURE 11
비밀값과 환경변수
11강 배포 실습에서는 API 없는 안전 버전을 기준으로 합니다.
  • API Key는 코드에 직접 쓰지 않는다
  • 공개 저장소에 비밀값을 올리지 않는다
  • 프론트엔드만 있는 정적 사이트는 키 보호가 어렵다
  • AI API 호출은 별도 서버나 안전한 백엔드 구조가 필요하다
  • 처음 배포는 API 없는 버전으로 제한한다

LECTURE 11
좋은 피드백의 조건
2강의 ‘여기를 이렇게’ 원칙이 다시 등장합니다.
  • 화면 위치가 구체적이다
  • 무엇이 불편했는지 말한다
  • 원하는 방향을 제시한다
  • 새 기능 요구와 버그를 구분한다
  • 바로 고칠 것과 나중에 할 것을 나눈다
L11 · PART 3

LECTURE 11
롤백을 생각하라
배포의 핵심은 앞으로만 가는 것이 아니라 돌아올 수 있는 것입니다.
  • 배포 후 문제가 생기면 이전 버전으로 돌아갈 수 있어야 한다
  • 수정 전 파일을 남겨둔다
  • 큰 변경은 한 번에 많이 하지 않는다
  • 동작하던 버전을 기준점으로 삼는다
  • 링크 공유 후에는 변경 내용을 기록한다
