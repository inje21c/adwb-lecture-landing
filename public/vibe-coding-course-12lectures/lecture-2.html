기능 vs 장면
기능 중심 표현장면 중심 표현
로그인 기능사용자가 앱을 열고 바로 시작할 수 있다
일정 기능이번 주 모임 날짜와 장소가 가장 먼저 보인다
참석 체크 기능참석·불참·미정 중 하나를 바로 누른다
통계 기능구역장이 참석 현황을 한눈에 본다
복사 기능보고용 문구를 바로 복사한다
AI는 장면을 이해하면 화면 흐름·버튼 위치·문구·정보 순서를 스스로 설계한다.

AI가 알아서 해도 되는 것은 [문구·배치·사용성 개선]이다. 하지만 [데이터 구조 변경, 신규 기능 추가, 외부 연동]은 바로 구현하지 말고 먼저 제안만 해라. 회사 고객정보·비밀번호·API Key 같은 민감정보는 입력하지 않는다. 이번에는 완성본이 아니라 1차 초안을 만들어라.

EXAMPLE — 구역모임 앱
✓ 좋은 프롬프트
나는 교회 구역모임 일정을 공유하고 참석 여부를 쉽게 수집하는 모바일 우선 PWA를 만들고 싶다. 현재 구역장은 매주 카카오톡으로 일정을 안내하고, 참석 여부를 일일이 확인한 뒤 다시 정리해서 보고해야 한다. 이 앱의 목적은 구역원의 응답을 쉽게 받고, 구역장의 정리 시간을 줄이는 것이다. 주요 사용자는 구역장과 구역원이다. 구역원은 IT에 익숙하지 않아도 이번 주 모임 정보를 보고 참석·불참·미정 중 하나를 쉽게 선택할 수 있어야 한다. 구역장은 참석 현황을 한눈에 보고 보고용 문구를 복사할 수 있어야 한다. 사용자가 앱을 열면 이번 주 모임 카드가 가장 먼저 보여야 한다. 카드에는 날짜, 시간, 장소, 안내문, 참석 여부 선택 버튼이 들어간다. 문구 개선·화면 배치·모바일 가독성 개선은 AI가 판단해서 반영해도 된다. 하지만 DB 구조 변경, 인증 방식 변경, 복잡한 관리자 기능, 외부 API 연동은 바로 구현하지 말고 먼저 제안만 해라. 실제 회사 고객정보·비밀번호·API Key 같은 민감정보는 입력하지 않는다. 이번에는 완성본이 아니라 1차 초안을 만들어라.

CASE — 개발자식 해석
개발자 남편에게 심부름시키기
부인의 부탁
"마트에 가서 빵 하나 사 와.
그리고 계란이 있으면 12개 사 와."
↓
결과
남편은 빵을 12개 사 왔다.
"계란이 있었거든."
두 가지 해석
의도
빵 1개. 계란 있으면 계란 12개.
논리적 해석
빵을 사라. 계란이 있으면 빵을 12개.
AI도 이렇게 해석할 수 있다.
맥락을 ‘넘겨짚는’ 위험이 항상 있다.

정확한 심부름 = 좋은 프롬프트

금지 사항다른 물건은 사지 마요청하지 않은 기능은 추가하지 마라
