# AI-Driven Working Backwards
## 보도자료에서 RFP까지 — AI 기반 IT 조달 요구사항 구체화 방법론

---

## Hero Section

**헤드라인:** 보도자료에서 RFP까지, 3~4일.
**서브헤드:** Amazon Working Backwards + AI 바이브코딩 = 현업이 직접 만드는 시각적 요구사항
**CTA:** 방법론 알아보기 | 강의 문의하기

---

## Section 1: 문제 정의 (Problem)

### IT 프로젝트의 80%는 기술이 아닌 요구사항에서 실패합니다.

현업은 자기가 원하는 것을 텍스트, 혹여 PPT 정도로밖에 표현하지 못합니다.

그 문서를 기획자가 듣고 → 문서로 만들고 → IT가 해석합니다.
3단계를 거치면서 원래 의도의 절반은 사라지고, 나머지 절반은 왜곡됩니다.

RFP는 텍스트로 가득 차 있고,
개발업체마다 이걸 다르게 읽고,
견적은 2배씩 차이가 나고,
개발이 끝나면 서로 "약속이 달랐다"고 합니다.

요구공학에서 말하는 전형적인 페인포인트들입니다.

---

## Section 2: 해법 (Solution)

### Amazon은 20년 전에 답을 알고 있었습니다.

"제품을 만들기 전에 보도자료를 먼저 써라." — Working Backwards

저는 여기에 AI를 결합했습니다.

**AI-Driven Working Backwards:**
보도자료에서 시작하여, AI 바이브코딩으로 목업을 만들고,
현업이 직접 체험하고 피드백하는 반복 사이클을 거쳐,
명확한 RFP까지 도달하는 12단계 파이프라인.

---

## Section 3: 12단계 워크플로우 전체 구조

### 영역 A — 목표 선명화 + 프롬프트 설계

| # | 단계 | 목적 | 주도 도구 |
|---|------|------|-----------|
| 1 | 보도자료 작성 | 고객가치 정의 | ChatGPT / Claude |
| 2 | KPI 구체화 | 성공 기준 정의 | ChatGPT / Claude |
| 3 | UI 생성 프롬프트 개발 | 시각화 설계도 확보 | ChatGPT + Behance/Dribbble |

### 영역 B — 반복적 구체화 (핵심 엔진)

| # | 단계 | 목적 | 주도 도구 |
|---|------|------|-----------|
| 4 | Lovable UI 생성 | 화면 가시화 | Lovable |
| 5 | GitHub 푸시·클론 | 코드 확보 | GitHub |
| 6 | 소스 정리 | 반복 가능 구조화 | Claude Code |
| 7 | 데모 기능 구현 | 실행 가능성 부여 | Codex + Claude Code |
| 8 | Firebase 배포 | 시연 가능 상태 | Firebase |
| 9 | 시연 및 피드백 | 요구사항 탐색 | 사람 + 데모 |
| 10 | 반복 수정 | 요구사항 수렴 | Claude Code |

> Step 9~10은 요구사항이 수렴될 때까지 반복됩니다.
> 이 반복 사이클이 방법론의 핵심 엔진입니다.

### 영역 C — 산출물 확정

| # | 단계 | 목적 | 주도 도구 |
|---|------|------|-----------|
| 11 | 설계서 작성 | 최종 구조 문서화 | Claude Code + ChatGPT |
| 12 | RFP 작성 | 발주 준비 | ChatGPT / Claude |

## Section 4: 핵심 포지셔닝

### 바이브코딩은 개발 도구만이 아닙니다.

엔터프라이즈 시스템을 처음부터 끝까지 바이브코딩만으로 개발하는 것은 현실적이지 않습니다.

그러나 현업의 생각을 '보이게' 만들고,
반복적으로 정제하여 명확한 RFP까지 도달하는 데 쓰면 —
**매우 효율적입니다.**

> 기술은 도구이고, 사람이 중심입니다.

## Section 5: 기존 방식 대비 비교

| 비교 항목 | 기존 방식 | AI-Driven Working Backwards |
|-----------|-----------|---------------------------|
| 요구사항 전달 | 현업→기획자→IT (3단계 왜곡) | 현업이 데모를 직접 보고 피드백 (1단계) |
| 1차 산출물 | 4~8주 | 3~4일 |
| 요구사항 정제 | 문서 검토 회의 (추상적) | 데모 시연→피드백→수정 반복 (구체적) |
| RFP 형태 | 텍스트 중심, 범위 모호 | 데모 URL + 목업 + 설계서 첨부 |
| 견적 편차 | 업체 간 2배 이상 차이 | 목업 기반 범위 명확, 편차 최소화 |
| 개발 후 CR | 빈번 ("원했던 게 아닌데") | 최소화 (데모로 사전 합의) |
| 비용 | 기획 컨설팅 수천만원 | AI 도구 月 수만원 + 내부 인력 |
